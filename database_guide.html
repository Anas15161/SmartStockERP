Architecture de Données : Oracle & JPA

Comment nous avons orchestré la persistance haute performance entre Jakarta EE 10 et Oracle XE.

01

Socle Technologique : Oracle XE 23c

Le projet repose sur Oracle Database 23c Free, configuré en mode Pluggable Database (PDB). Ce choix garantit une compatibilité totale avec les standards industriels.

Configuration de connexion :
  • Service Name : XEPDB1 (Instance Pluggable)
  • Driver : Oracle JDBC Thin Driver (ojdbc11)
  • Dialect : OracleDialect (Optimisé pour Hibernate 6)
02

Configuration de l'Unité de Persistance

Le fichier persistence.xml est le cœur de la communication. Il définit comment JPA doit se comporter avec Oracle et Hibernate.

src/main/resources/META-INF/persistence.xml
<persistence-unit name="SmartStockPU" transaction-type="RESOURCE_LOCAL">
    <provider>org.hibernate.jpa.HibernatePersistenceProvider</provider>
    
    <properties>
        <!-- Connection Oracle -->
        <property name="jakarta.persistence.jdbc.url" value="jdbc:oracle:thin:@localhost:1521/XEPDB1"/>
        <property name="jakarta.persistence.jdbc.user" value="smartstock"/>
        
        <!-- Hibernate Magic -->
        <property name="hibernate.dialect" value="org.hibernate.dialect.OracleDialect"/>
        <property name="hibernate.hbm2ddl.auto" value="update"/> <!-- Synchro Schema Auto -->
        <property name="hibernate.show_sql" value="true"/>
    </properties>
</persistence-unit>
03

Mapping Objet-Relationnel (ORM)

Nous transformons des tables Oracle froides en objets Java vivants via les annotations Jakarta Persistence.


SUPPLIERS
ID, NAME, EMAIL

PRODUCTS
ID, LABEL, SUPP_ID
Relation Many-to-One : Chaque produit est lié à un fournisseur. JPA gère automatiquement la Foreign Key (FK) dans la table PRODUCTS, évitant ainsi des jointures SQL manuelles complexes.
Mapping de la relation dans Product.java
@ManyToOne(fetch = FetchType.EAGER)
@JoinColumn(name = "supplier_id")
private Supplier supplier;
04

Gestion via l'EntityManager

L'EntityManager est notre interface unique pour toutes les opérations. Nous avons encapsulé sa logique dans un Repository générique pour maximiser la réutilisation du code.

Repository Générique (Extrait CRUD)
public T update(T entity) {
    EntityTransaction tx = em.getTransaction();
    tx.begin();
    T merged = em.merge(entity); // Synchronise avec Oracle
    tx.commit();
    return merged;
}
Le cycle de vie : Hibernate suit l'état de l'objet (Transient, Managed, Detached). Une fois "Managed" par l'EntityManager, toute modification sur l'objet Java est automatiquement répercutée dans la base Oracle lors du commit.
05

Audit & Cycle de Vie (Callback JPA)

Pour assurer la traçabilité des données sous Oracle, nous utilisons les Lifecycle Callbacks.

Audit Automatique
@PrePersist
protected void onCreate() {
    this.createdAt = LocalDateTime.now(); // Avant insertion Oracle
}

@PreUpdate
protected void onUpdate() {
    this.updatedAt = LocalDateTime.now(); // Avant mise à jour Oracle
}
